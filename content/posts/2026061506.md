## 避坑设置二：合理的超时和重试策略

很多新手在调用API时根本不设置超时，或者把超时时间设得太短（比如5秒）。GPT-5.1作为大型模型，在处理复杂任务或推理时，响应时间可能超过10秒，甚至15秒。如果你设了个5秒的时限，请求大概率会失败。

**正确做法：** 设置合理的超时参数，并实现指数退避重试逻辑。

python
import time
from openai import OpenAI

client = OpenAI(
    base_url="https://www.yunwuai.cc/v1",
    api_key="your-api-key-here",
    timeout=60.0  # 建议设置为30-60秒
)

def try_api_with_retry(max_retries=3):
    for attempt in range(max_retries):
        try:
            response = client.chat.completions.create(
                model="gpt-5.1",
                messages=[{"role": "user", "content": "Explain quantum computing in 100 words."}],
                temperature=0.7
            )
            return response
        except Exception as e:
            if attempt < max_retries - 1:
                wait_time = 2 ** attempt  # 指数退避：2, 4, 8秒
                print(f"请求失败，{wait_time}秒后重试。错误：{e}")
                time.sleep(wait_time)
            else:
                raise e


这个设置特别关键，尤其是在网络波动时，能自动处理临时故障，保证你的应用不掉链子。

## 避坑设置三：合理控制并发与速率限制

新手常常犯的另一个错误：一上来就开几十个并发请求，结果触发了API的速率限制(Rate Limit)，被返回429错误。GPT-5.1的API有严格的频率和并发限制，超过就会拒绝服务。

**正确做法：** 使用连接池，并主动控制并发数。同时，建议开启“流式输出”(Streaming)来处理长回复，避免一次性拉取超时。

python
from openai import OpenAI

client = OpenAI(
    base_url="https://www.yunwuai.cc/v1",
    api_key="your-api-key-here",
    max_retries=3       # 自动重试
)

# 使用流式输出应对长文本，避免长时间等待
response = client.chat.completions.create(
    model="gpt-5.1",
    messages=[{"role": "user", "content": "请写一篇3000字的文章"}],
    stream=True          # 启用流式响应
)

for chunk in response:
    if chunk.choices[0].delta.content:
        print(chunk.choices[0].delta.content, end="")


**控制并发示例：**

python
from concurrent.futures import ThreadPoolExecutor, as_completed
import threading

# 使用线程锁控制并发数，建议全局并发不超过5个
concurrency_limit = threading.Semaphore(5)

def safe_api_call(messages):
    with concurrency_limit:
        return client.chat.completions.create(
            model="gpt-5.1",
            messages=messages
        )

with ThreadPoolExecutor(max_workers=5) as executor:
    futures = [executor.submit(safe_api_call, [{"role": "user", "content": f"问题{i}"}]) for i in range(10)]
    for future in as_completed(futures):
        result = future.result()
        print(result.choices[0].message.content)


这样既能保证速度，又不会因为过载被封，确保调用稳定。
