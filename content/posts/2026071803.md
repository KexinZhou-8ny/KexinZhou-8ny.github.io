## 它解决的到底是什么问题

一句话说清楚：这套示例让你**省下80%的调试时间**，专注于业务逻辑，而不是框架配置。

这里说的“调试时间”，指的是那些让开发者头疼的常见痛点：

- **模块耦合**：集成Spring后，你的代码立即和Spring生态绑定，改动或迁移时需要重写大量配置。

这套示例的核心思路是**极简主义**：用尽量少的代码，做尽量多的事。所有逻辑封装在一个Java类里，复制粘贴后直接运行，不需要理解任何框架原理。

## 示例代码到底有多短

看看这个例子，你能直接粘贴到你的IDE里运行：

java
import com.google.gson.Gson;
import com.google.gson.JsonObject;
import java.net.URI;
import java.net.http.HttpClient;
import java.net.http.HttpRequest;
import java.net.http.HttpResponse;
import java.time.Duration;

public class SimpleErnieDemo {
    // ERNIE API 调用地址（替换为你自己的API Key）
    private static final String API_URL = "https://www.yunwuai.cc/v1/chat/completions";
    private static final String API_KEY = "sk-your-api-key"; // 替换为你的密钥

    public static void main(String[] args) throws Exception {
        HttpClient client = HttpClient.newBuilder()
                .connectTimeout(Duration.ofSeconds(10))
                .build();

        JsonObject requestBody = new JsonObject();
        requestBody.addProperty("model", "ernie-4.0-8k");
        JsonObject message = new JsonObject();
        message.addProperty("role", "user");
        message.addProperty("content", "用一句话介绍[云雾ai大模型聚合站](https://www.yunwuai.cc/)。");
        requestBody.add("messages", new com.google.gson.JsonArray());
        requestBody.getAsJsonArray("messages").add(message);

        HttpRequest request = HttpRequest.newBuilder()
                .uri(URI.create(API_URL))
                .header("Authorization", "Bearer " + API_KEY)
                .header("Content-Type", "application/json")
                .timeout(Duration.ofSeconds(30))
                .POST(HttpRequest.BodyPublishers.ofString(requestBody.toString()))
                .build();

        HttpResponse<String> response = client.send(request, HttpResponse.BodyHandlers.ofString());
        System.out.println("响应状态码：" + response.statusCode());
        System.out.println("响应内容：" + response.body());
    }
}


这段代码仅有35行（含注释），实现了完整的ERNIE API调用流程。你将`API_KEY`替换为你在[云雾ai大模型聚合站](www.yunwuai.cc)申请的密钥，就能直接运行。

## 性能对比：为什么不用Spring

很多人习惯用Spring RestTemplate来完成HTTP请求。但这会带来几个隐藏问题：

- **启动慢**：Spring Boot应用需要大约1-2秒才能启动，而纯命令行的Java程序启动时间不到0.1秒。
- **依赖链路长**：Spring RestTemplate依赖Jackson（用于JSON解析）、Spring Core、Spring Context，加起来至少有5-8个Jar包。我们的示例只依赖`gson`这一个包。
- **调试体验差**：Spring的异常堆栈往往包含大量框架内部逻辑，排查问题时需要先看日志、再查文档，过程非常繁琐。

下面的表格能更直观地对比两种方案：

| 评估维度           | 传统Spring方案                                                                 | 轻量级Java单线程方案                                                                 |
| ------------------ | ------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------ |
| **第三方依赖**     | 至少5个（Spring Boot、Jackson、Logback等）                                     | 1个（gson）                                                                          |
| **启动时间**       | ~1-2秒（加载容器、扫描Bean）                                                    | 0.1-0.3秒（直接启动）                                                                |
| **开发耗时**       | 约30分钟（编写配置、排查依赖冲突）                                               | 约5分钟（复制粘贴、改API Key）                                                        |
| **集成第三方工具** | 与Spring生态耦合，迁移时需重写                                                 | 无耦合，任何工具都可轻松集成                                                         |
| **环境适配**       | 必须安装Maven/Gradle，需要IDE支持                                               | 仅需JDK 11+，记事本都能运行                                                          |

对于快速原型验证（POC）、技术预研或轻量级工具场景，轻量级方案显然更省心。
