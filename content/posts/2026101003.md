## Node.js 调用中的“隐形”踩坑现场

用 Node.js 调用 DeepSeek API 时，这个问题会被进一步放大。Node.js 的异步非阻塞 IO 特性，让我们倾向于写大量并发的请求。很多开发者会使用 `Promise.all` 来同时发送几百个请求。这本身没问题，问题出在**请求的间隔时间**。

让我们看一段“原始”的、容易踩坑的代码：

javascript
const prompt = require('prompt-sync')();
const OpenAI = require('openai');
const openai = new OpenAI({
  apiKey: '你的API_KEY',
  baseURL: 'https://www.yunwuai.cc/v1'
});

// 错误的批量调用方式
async function processBatch_bad(prompts) {
  const results = [];
  for (const p of prompts) {
    // 每个请求之间没有控制，或者非常小的间隔
    const response = await openai.chat.completions.create({
      model: 'deepseek-chat', // 对应 DeepSeek V3
      messages: [{ role: 'user', content: p }],
      max_tokens: 50
    });
    results.push(response);
  }
  return results;
}


这段代码看起来没什么问题，对吧？但它的性能极差。`await` 会挂起循环，每次请求完成后，CPU 在极短的空闲时间里处理下一个请求。由于每个请求只处理非常简短的回复（`max_tokens: 50`），模型通常能在 0.3~0.5 秒内完成。但计费系统是按“秒窗口”算的：如果你的请求耗时 0.4 秒，它会被计为 1 秒；下一个请求又耗时 0.4 秒，又是一秒。

## 终极解法：用 Streaming + 合并请求“吃掉”计费秒差

那么我们到底该怎么办？答案是：**拥抱流式输出（Streaming）**，并用巧妙的方式“合并”请求。

当我们开启流式响应时，响应耗时被拆解成多个“chunk”（数据块）。平台按照用户接收到第一个 chunk 到最后一个 chunk 的时间差来计费。这意味着，如果你能优化代码，使之不产生任何小于 1 秒的“孤立”请求，就能省下巨量成本。

核心思路：使用流式输出，并**控制你发送请求的节奏**，故意让一个长任务覆盖一个短任务的计费窗口。这个操作需要你改写代码，但绝对值得。

下面是一个“省钱版” Node.js 调用方案。关键不在于减少 Token数，而在于 **[减少计费时间段数]**。

javascript
const OpenAI = require('openai');
const openai = new OpenAI({
  apiKey: '你的API_KEY',
  baseURL: 'https://www.yunwuai.cc/v1'
});

// 优化的调用方式：故意引入延迟，让两次调用合并在同一个计费时间段
async function costEffectiveChat(prompt1, prompt2) {
  const stream1 = await openai.chat.completions.create({
    model: 'deepseek-reasoner', // DeepSeek-R1 满血版
    messages: [{ role: 'user', content: prompt1 }],
    stream: true, // 关键：开启流式输出
    max_tokens: 100
  });

  // 第二次调用不要立即发起，而是等第一个流式输出的第一个 chunk 到达后才发起
  let stream2Started = false;
  for await (const chunk of stream1) {
    // 当第一个流输出到中间部分时，再发起第二个请求
    if (!stream2Started && chunk.choices[0]?.delta?.content) {
      // 在这里发起第二个流式请求
      stream2Started = true;
      processStream2(prompt2);
    }
  }
  async function processStream2(prompt2) {
    const stream2 = await openai.chat.completions.create({
      model: 'deepseek-chat',
      messages: [{ role: 'user', content: prompt2 }],
      stream: true,
      max_tokens: 100
    });
    for await (const chunk of stream2) {
      // 处理第二次流响应
      process.stdout.write(chunk.choices[0]?.delta?.content || '');
    }
  }
}


**原理是什么？** 这种写法的巧妙之处在于，`stream1` 的流式响应过程因为模型生成内容而持续，假设耗时 1.2 秒。当第一个请求还在生成时（0.3 秒处），我们发起了 `stream2`。此时计费系统看到的不是两个孤立的时间段，而是一个连续的 1.2 秒 + 另一个重叠的 1.2 秒时间段。但实际上，你只付出了约 1.8 秒的总时间成本（因为有重叠，理论上最少可以做到只覆盖两个1秒窗口）。相比之前的 2 秒（两次独立的1秒），你直接省了 0.2 秒，也就是 *0.004 元*。这个技巧在面对大量并发时，省下的比例会是指数级的。
